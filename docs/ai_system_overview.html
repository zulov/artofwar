1. System Overview — One Tick
Every AI tick: Master Brain reads game state and outputs urgencies. Sub-brains generate specific requests. Want List prioritizes and executes. Lacking feedback loops back.
2. Master Brain — Strategic Coordinator
Reads 34 game-state inputs and outputs 9 urgency signals. These urgencies drive all sub-brains and the want list.

3. Economy Brain — Resource Management
Receives master urgencies + lacking feedback. Outputs worker counts, resource priorities, building needs, worker reassignment, and upgrade urgencies. 38 inputs -> 19 outputs.

Key Outputs (19)

  • workerAllocation — drives workerCount = round(workerAllocation * 3), max 3 workers added to want list per tick (threshold 0.1)
  • foodPriority / woodPriority / stonePriority / goldPriority — which resource to collect first
  • expandPriority — territorial expansion weight
  • reassignWorkers — should we pull a worker off the lowest-priority resource (threshold: 0.5)
  • 10 building need scores — NEED_BONUS_FOOD/WOOD/STONE/GOLD, NEED_FOOD_SOURCE, NEED_FOOD_STORAGE, NEED_GOLD_STORAGE, NEED_GOLD_REFINE, NEED_STONE_REFINE, NEED_WOOD_SOURCE. Matched to resource buildings via DB fields (top 3 needs > 0.1 become BUILDING wants)
  • workerUpgradeUrgency / resBuildingUpgradeUrgency — drive UNIT_UPGRADE / BUILDING_UPGRADE wants (threshold 0.1)

Worker Collection Flow

Free workers are split across the 4 resource types proportionally to their (non-negative) priorities (largest-remainder rounding). Each worker takes the highest-priority resource that still needs workers and has a reachable node (expanding resource-query levels). If reassignment is active and the priority gap > 0.3, one busy worker is stolen from the worst resource. Only one worker is reassigned per tick.

4. Unit Brain — Profile-Based Unit Selection
Instead of picking a unit by ID, outputs a 23-dimensional "ideal unit profile" describing what the AI wants. Matched against real units by distance. 21 inputs -> 23-dim profile + 2 upgrade urgencies.
5. Metric System — 23-Dimensional Unit Space
Each unit type has a precomputed 23-float metric vector. The brain outputs a vector in the same space. Distance = similarity.

Validation

All metric values are asserted to be in [0, 1] at DB load time. If a new unit exceeds this range, the assert fires, forcing the designer to either adjust the unit stats or increase the normalization weight. This prevents silent metric-brain mismatch.

6. Distance Matching — How Units Are Chosen
Sum of squared errors between brain output and unit metric vectors. Lower distance = more similar = higher selection probability.
7. Want List — Priority Queue Execution
All production requests compete in a single priority queue. Multiple items can execute per tick.
8. Military Brain — Combat Orders
Runs in action() (feeds composition prefs to Unit Brain); army movement is issued in order(). Direct army orders, no resource cost. 27 inputs -> 24 outputs.

Outputs (24)

  • 21 center-pair pressures — one signed value per unordered pair of the 7 "centers" (OUR_ARMY, OUR_ECON, OUR_BUILDING, ENEMY_ARMY, ENEMY_ECON, ENEMY_BUILDING, LAST_BATTLE). Sign encodes the pull direction between the two centers.
  • preferInfantry / preferRange / preferCavalry — army composition preferences fed into the Unit Brain as PREFER_* inputs.

Decision Flow (order())

  • 1. Snapshot centers: influence-map centers of mass for the 7 center types (own + enemy army/econ/building + last battle).
  • 2. Per-unit target: MilitaryCommandCalculator scores each unit's position against the center-pair pressures and picks the highest-pressure target (attack/defend/move-to-center). Units below MIN_ARMY_ORDER_PRESSURE (0.1) hold.
  • 3. Position: for attack buckets, AttackSpatialBrain weights (6 influence-map layers) pick the best visible area via getBestVisibleAreas(); fallback is the target center.
  • 4. Execution: far units GO toward the target (carrying their pressure as command priority); units within ~30 ATTACK the closest enemy. Idle hold units only engage nearby enemies.

Limitations

  • No retreat — army fights to the death, no health threshold check
  • Center-position fallback ignores fog of war (can advance toward stale/unseen centers)
  • Single enemy only — ignores other players in FFA
9. History System — Feedback Loop
128-entry circular buffer tracking actions (build, produce) and orders (attack, defend, collect). Provides recency and failure scores to Master Brain.

Two Buffers

BufferEntryTracks
Actions (128)tick, AiActionType, AiActionResult, chosenIdCREATE_UNIT, CREATE_WORKER, CREATE_BUILDING_*
Orders (128)tick, AiOrderType, AiOrderResult, unitCountATTACK_*, DEFEND_*, COLLECT_RESOURCE_*

Scoring Functions

  • recencyScore(type, lookback) — sum of (1 - age/lookback) for successful entries. Higher = more recent activity
  • failureScore(type, lookback) — same formula but for failed entries. Higher = more recent failures
  • Composite: buildingFailureScore, unitFailureScore, attackActivityScore, defendActivityScore

Lookback is 900 ticks (~30 seconds at 30 tps). With 128 entries, the buffer covers sustained activity without overflowing during active phases.

10. Worked Examples

Example 1: Early Game — Economy Focus

Tick 300. Score: 50. Army: 2 militia. Enemy: unknown (no contact). Resources: 200F, 150W, 50S, 0G. 5 workers.

Master Brain: Low enemy threat -> low MILITARY_URGENCY (-0.3). Few buildings -> BUILDING_URGENCY = 0.6. Low resources -> ECONOMY_URGENCY = 0.8. WORKER_URGENCY = 0.7.

Economy Brain: workerAllocation = 0.7 -> workerCount = round(0.7 * 3) = 2. foodPriority = 0.8, woodPriority = 0.6, stonePriority = 0.1, goldPriority = 0.0. NEED_FOOD_SOURCE = 0.7 (need a food bonus building).

Unit Brain: unitUrgency = 0.1 -> count = round(0.1 * 5) = 1. Profile outputs weak preference for cheap infantry. One unit added to want list.

Want List: Workers (priority 0.7 x2) vs food building (0.7) vs Infantry unit (0.1), sorted by priority and executed top-down while affordable. Workers and the resource building queue first; the infantry may not have enough resources left.

Result: ~2 workers produced, 1 resource building built, 0-1 military units. Economy grows.

Example 2: Mid Game — Enemy Cavalry Rush

Tick 2000. Enemy has 15 cavalry, 5 archers. Our army: 8 infantry, 3 archers. KD_RATIO = 0.3 (losing).

Master Brain: High enemy army + low KD -> UNIT_URGENCY = 0.9, MILITARY_URGENCY = 0.7, ATTACK_URGENCY = -0.2 (defend). DEFENCE_BUILDING_URGENCY = 0.5.

Unit Brain: count = round(0.9 * 5) = 5. Sees enemy cavalry ratio is high.

Brain outputs: TYPE_INFANTRY = 0.8, TYPE_HEAVY = 0.3, BONUS_CAVALRY = 0.9, TYPE_CAVALRY = -0.7 (avoid cavalry).

Profile Matching: lowestWithRand runs 5 times. Spearman (anti-cavalry infantry) has lowest distance. Results: 4x Spearman, 1x Pikeman (similar but slightly different roll).

Military Brain: Center-pair pressures pull the army toward OUR_BUILDING (defend the base) over enemy centers. Per-unit, MilitaryCommandCalculator routes units to defend near own buildings.

Result: 5 anti-cavalry units queued (if affordable). Army defends base. Counter-composition evolves naturally.

Example 3: Unbuildable Unit Fallback

Brain wants cavalry archers, but the Archery Range is destroyed. Only Barracks is standing.

resolveUnit(): Profile matches "Mounted Archer" (cavalry + range type). Returns it.

executeUnit(): getBuildingsCanDeploy("Mounted Archer") returns empty. Sets pendingLackingBuilding = Archery Range ID.

Feedback: Next tick, lackingBuildingForUnit is set. submitBuildingRequest adds Archery Range to want list with boosted urgency (min 0.5). Meanwhile, the WantItem for Mounted Archer fails and decays.

Result: AI prioritizes rebuilding the Archery Range. Unit request fails gracefully. Resources freed for the building.

Example 4: Negative Profile = Active Avoidance

Brain wants siege units to break enemy walls.

Brain outputs: BONUS_BUILDING = 0.8, TYPE_HEAVY = 0.5, TYPE_CAVALRY = -0.9, TYPE_LIGHT = -0.7, ATTACK = 0.6

Distance calculation for a Light Scout:

TYPE_CAVALRY: (-0.9 - 1.0)^2 = 3.61, TYPE_LIGHT: (-0.7 - 1.0)^2 = 2.89, BONUS_BUILDING: (0.8 - 0.0)^2 = 0.64

Total penalty from just 3 dimensions: 7.14

Distance for a Battering Ram:

TYPE_CAVALRY: (-0.9 - 0.0)^2 = 0.81, TYPE_LIGHT: (-0.7 - 0.0)^2 = 0.49, BONUS_BUILDING: (0.8 - 0.67)^2 = 0.02

Total: 1.32

Result: Battering Ram wins decisively. The [-1, 0] range provides 3x stronger rejection than [0, 1] alone.

11. Pros and Cons

Strengths

  • Nation-agnostic unit selection — Same NN works across factions with different unit rosters. Profile describes properties, not IDs.
  • Emergent counter-play — The brain learns to output anti-cavalry profiles when enemy has cavalry, without hardcoded rules.
  • GA-evolvable — All brains are neural networks with weights from CSV files. No hand-tuning needed for strategy. Evolution discovers good play.
  • Stochastic variety — lowestWithRand prevents perfectly predictable play. Multiple rolls per tick produce army variety.
  • Priority-based resource allocation — Want list naturally balances economy vs military vs tech. Persistent needs escalate via age boost.
  • Unit & building upgrades — Unit Brain and Economy Brain emit upgrade urgencies; UNIT_UPGRADE / BUILDING_UPGRADE wants compete in the same queue. Tech level (avg upgrade level) feeds back into Master/Unit/Military brains.
  • Feedback loops — Lacking resources feed back to Master Brain. Building failures feed back to unit production. History informs future decisions.
  • Metric validation — Assert at DB load ensures all metrics stay in [0, 1]. New units that violate this are caught immediately.
  • Negative outputs as repulsion — tanh [-1, 1] gives the brain 3x stronger "avoid" signals vs [0, 1] range, enabling active trait avoidance.
  • Efficient multi-sampling — lowestWithRand(diffs, count) computes weights once, samples N times. sampleWeighted is O(candidates) per sample.
  • Brain caching — BrainProvider caches brains by filename. Same-input same-output optimization in Brain::decide() skips redundant forward passes.

Weaknesses

  • No retreat logic — Army fights to death. No health threshold, no "pull back and regroup." Armies are wasted in losing engagements.
  • Non-resource building selection is a stub — resolveBuilding() for DEFENCE/TECH/UNITS/OTHER returns buildings.at(0). No NN-driven selection (TODO in code).
  • No scouting — AI cannot explore fog of war. Plays blind until enemy contact. Disadvantaged on large maps.
  • Single enemy — getEnemyFor() returns one enemy. FFA/multiplayer ignored.
  • No attack outcome tracking — History records "order sent" as success, not whether the battle was won.
  • Worker reassignment is slow — Only 1 worker per tick, only if priority gap > 0.3. Slow to adapt economy.
  • Save/load breaks deltas — MasterBrain prev* fields not serialized. First tick after load produces false delta spikes.
  • No training pipeline in repo — No GA implementation, no fitness functions, no weight export. Brains must be evolved externally (see tools/gen_brain_csv.ps1 for untrained weight generation only).
  • lowestWithRand samples with replacement — Can return the same unit type multiple times when variety might be better.
