EconomyBrain Architecture - Unified

Single EconomyBrain handles macro economy + resource building decisions. No sub-brain.

Design Change Log

Previous: Option B (Two-Tier)
  • EconomyBrain (24 inputs, 8 outputs) + ResourceBrain sub-brain (20 inputs, 9 outputs)
  • ResourceBrain activated when RESOURCE_BUILDING_URGENCY > threshold
  • ResourceBrain output: 5 capability needs (BONUS/SPAWN/CONVERT/STORAGE/REFINE) + 4 resource preferences
  • Orchestrator combined capability + resource preference via argmax to pick building
Current: Unified (Single Brain)
  • EconomyBrain (38 inputs, 18 outputs) does everything, including upgrade urgencies
  • ResourceBrain removed entirely
  • 10 direct building-type need outputs (e.g. NEED_BONUS_FOOD, NEED_FOOD_SOURCE, NEED_FOOD_STORAGE)
  • Orchestrator scores resource buildings against these needs via DB fields

Why the change: ResourceBrain was a middleman. EconomyBrain already had most of the same inputs (gather speeds, stockpiles, storage ratios). The capability+resource separation (5+4 outputs) was indirect -- the orchestrator had to combine them, and some combinations didn't map to any real building. Direct building-type outputs eliminate ambiguity.

Architecture Overview

Main Brain Unified

EconomyBrain is a single neural network that handles all economy decisions: worker training, resource allocation, resource building selection, and worker/resource-building upgrades. It receives 38 inputs covering scores, resources, gather speeds, workers, storage/refine state, bonus coverage, nearby supply, building counts, lacking feedback, urgencies from MasterBrain, collection-failure history, average levels, and game time. It outputs 18 values.

Building Resolution Algorithm

AiOrchestrator::createResBuilding()

Each tick the orchestrator scores every resource building the nation can build against the 10 need outputs and submits the most-needed ones as BUILDING wants. No hardcoded building IDs -- matching uses DB fields from db_building and the player's current db_building_level.

Algorithm (AiOrchestrator::createResBuilding)
1. For each resource building b the nation can build:
   need = 0
   if b is a FOOD bonus building  (collect>0, resRange>0, resType==FOOD):  need = max(need, NEED_BONUS_FOOD)
   if b is a WOOD bonus building:                                          need = max(need, NEED_BONUS_WOOD)
   if b is a STONE bonus building:                                         need = max(need, NEED_BONUS_STONE)
   if b is a GOLD bonus building:                                          need = max(need, NEED_BONUS_GOLD)
   if b is a food source  (toResource>=0, spawnResourceRange<=0):          need = max(need, NEED_FOOD_SOURCE)
   if b is a wood source  (toResource>=0, spawnResourceRange>0):           need = max(need, NEED_WOOD_SOURCE)
   if b has foodStorage>0:                                                 need = max(need, NEED_FOOD_STORAGE)
   if b has goldStorage>0:                                                 need = max(need, NEED_GOLD_STORAGE)
   if b has stoneRefineCapacity>0:                                         need = max(need, NEED_STONE_REFINE)
   if b has goldRefineCapacity>0:                                          need = max(need, NEED_GOLD_REFINE)
   if need > 0.1: add candidate {need, b.id}
2. Keep the top 3 candidates by need (partial sort)
3. For each kept candidate: wantList.addRequest(BUILDING, need, b.id)
   -> the WantList then places affordable ones via BuildSpatialBrain

Resource-building upgrades use a separate path (resolveResBuildingUpgrade): candidate resource buildings with a next level are weighted by their resource-type priority plus matching subtype needs, then one is chosen by weighted random sampling.

Decision Examples

Example 1: Workers gathering wood without bonus

State: 5 workers on wood, no bonus building, BONUS_COVERAGE_WOOD = 0.0, GATHER_WOOD = 4.2/s

Brain sees: High wood gathering but zero bonus coverage. RES_WO_BONUS is high.

Brain outputs: NEED_BONUS_WOOD = 0.85 (highest)

Orchestrator: The wood bonus building scores need 0.85 (> 0.1) and lands in the top 3, so it is submitted as a BUILDING want and placed near unbonused wood nodes.

Example 2: Food stockpile exceeding storage

State: 480 food, foodStorage = 220, FOOD_STORAGE_RATIO = 2.18

Brain sees: FOOD_STORAGE_RATIO > 1.0, food being wasted past storage.

Brain outputs: NEED_FOOD_STORAGE = 0.9 (highest)

Orchestrator: A building with foodStorage > 0 scores 0.9 and is submitted as a BUILDING want (+food storage).

Example 3: Berry bushes running out, need renewable food

State: Only 2 berry bushes left, food stockpile = 60 falling, GATHER_FOOD = 1.2/s, LACKING_FOOD high

Brain sees: Low food, low gather speed, lacking-food feedback rising.

Brain outputs: NEED_FOOD_SOURCE = 0.95 (highest)

Orchestrator: A food-source building (toResource >= 0, spawnResourceRange <= 0) matches and is submitted as a BUILDING want.

Example 4: Trees depleted, need a wood source

State: Only 3 trees left, BONUS_COVERAGE_WOOD = 0.8 (existing trees already bonused)

Brain sees: High bonus coverage but trees running out; NEARBY_WOOD_SUPPLY falling.

Brain outputs: NEED_WOOD_SOURCE = 0.9 (highest), NEED_BONUS_WOOD = 0.1 (low, already covered)

Orchestrator: A wood-source building (toResource >= 0, spawnResourceRange > 0) matches and is submitted.

Example 5: Gold refinery needed

State: 6 workers on gold, GATHER_GOLD = 6.0/s, no refine capacity, GOLD_REFINE_GAP = 6.0 (gathering beyond refine)

Brain sees: High gold gather rate but GOLD_REFINE_RATIO = 0 (no refineries).

Brain outputs: NEED_GOLD_REFINE = 0.8 (highest)

Orchestrator: A building with goldRefineCapacity > 0 matches and is submitted as a BUILDING want.

Pros and Cons

Unified vs Two-Tier (Previous Option B)

Pros

  • No middleman: EconomyBrain diagnoses the problem AND picks the solution. No lossy signal passing between brains.
  • One brain to train: Single NN, single CSV, single training pipeline. No activation threshold to tune.
  • No duplicated inputs: ResourceBrain needed gather speeds, stockpiles, storage ratios -- all already in EconomyBrain.
  • No ambiguous combinations: Old approach output capability + resource preference independently. Some combos (e.g. REFINE_FOOD, SPAWN_STONE) don't map to any building. Brain could waste effort on impossible outputs.
  • Direct and debuggable: Each output maps to exactly one building type. Easy to inspect what the brain wants.
  • Fallback behavior: If nation doesn't have a building, that need is simply ignored (no candidate matches). No wasted signal.

Cons

  • Larger brain: 38 inputs x 18 outputs vs 24x8 + 20x9. More weights to train, slightly slower inference.
  • Not extensible to new combos: Adding a new building type (e.g. STORAGE_STONE) requires adding a new output to the enum and retraining. Two-tier approach would handle it automatically via capability + resource combination.
  • Building-type coupling: Output names reference specific building types. If game design changes significantly (e.g. removing farms), the enum needs updating.
  • No separation of concerns: Economy macro decisions and building micro decisions are mixed in one brain. May be harder to debug if the brain makes bad economy decisions AND bad building decisions simultaneously.

Direct Building Needs vs Capability + Resource (Alternative)

Pros of Direct Needs (current)

  • Unambiguous: Every output maps to exactly one building capability/resource. No invalid combinations.
  • Simpler orchestrator: Per-building max over matching needs + DB field matching. No capability*resource matrix to resolve.
  • Brain can learn building-specific patterns: E.g. "a food source is good when berries run out" is a different pattern from "a food bonus building is good when workers lack bonus". These don't share a common "FOOD building needed" signal.

Cons of Direct Needs

  • More outputs: 10 vs 9 (5 capabilities + 4 resource prefs). Grows if new buildings are added.
  • Redundancy: NEED_BONUS_FOOD/WOOD/STONE/GOLD are all "BONUS for resource X". The brain must independently learn the same pattern (low bonus coverage) for each resource.
  • Future nations: If a nation has different resource buildings, the output enum stays the same but the matching relies on DB capability fields rather than specific building identity.
