EconomyBrain Architecture - Unified

Single EconomyBrain handles macro economy + resource building decisions. No sub-brain.

Design Change Log

Previous: Option B (Two-Tier)
  • EconomyBrain (24 inputs, 8 outputs) + ResourceBrain sub-brain (20 inputs, 9 outputs)
  • ResourceBrain activated when RESOURCE_BUILDING_URGENCY > threshold
  • ResourceBrain output: 5 capability needs (BONUS/SPAWN/CONVERT/STORAGE/REFINE) + 4 resource preferences
  • Orchestrator combined capability + resource preference via argmax to pick building
Current: Unified (Single Brain)
  • EconomyBrain (34 inputs, 18 outputs) does everything
  • ResourceBrain removed entirely
  • 10 direct building-type need outputs (e.g. NEED_MILL, NEED_FARM, NEED_GRANARY)
  • Orchestrator does argmax over 10 needs, matches to building via DB fields

Why the change: ResourceBrain was a middleman. EconomyBrain already had most of the same inputs (gather speeds, stockpiles, storage ratios). The capability+resource separation (5+4 outputs) was indirect -- the orchestrator had to combine them, and some combinations didn't map to any real building. Direct building-type outputs eliminate ambiguity.

Architecture Overview

Main Brain Unified

EconomyBrain is a single neural network that handles all economy decisions: worker training, resource allocation, worker reassignment, and resource building selection. It receives 34 inputs covering scores, resources, gather speeds, workers, storage/refine state, bonus coverage, building counts, lacking feedback, and urgencies from MasterBrain. It outputs 18 values.

Building Resolution Algorithm

resolveBuilding(RESOURCE)

When the WantList executes a RESOURCE building request, resolveBuilding() picks the best building using the 10 need outputs. No hardcoded building IDs -- matching uses DB fields from db_building and db_building_level (level 0).

Algorithm (AiOrchestrator::resolveBuilding)
1. candidates = getBuildingsInType(RESOURCE)     // all nation's resource buildings
2. Sort 10 building needs by score descending
3. For each need (highest first):
   a. Skip if need <= 0
   b. For each candidate building:
      - Get level 0 from DB
      - Check: resourceType matches? (for BONUS types)
      - Check: capability matches?
        BONUS:   collect > 0 AND resourceRange > 0
        CONVERT: toResource >= 0 AND spawnResourceRange <= 0
        SPAWN:   toResource >= 0 AND spawnResourceRange > 0
        STORAGE: foodStorage > 0 (granary) or goldStorage > 0 (bank)
        REFINE:  stoneRefineCapacity > 0 or goldRefineCapacity > 0
      - If all criteria match: return this building
4. If no match found: return nullptr

Decision Examples

Example 1: Workers gathering wood without bonus

State: 5 workers on wood, no sawmill, BONUS_COVERAGE_WOOD = 0.0, GATHER_WOOD = 4.2/s

Brain sees: High wood gathering but zero bonus coverage. RES_WO_BONUS is high.

Brain outputs: NEED_SAWMILL = 0.85 (highest), RESOURCE_BUILDING_URGENCY = 0.7

Orchestrator: argmax = NEED_SAWMILL. Scans buildings: sawmill has collect > 0, resourceRange > 0, resourceType == 1 (WOOD). Match. Builds sawmill near unbonused wood nodes.

Example 2: Food stockpile exceeding storage

State: 480 food, foodStorage = 220, FOOD_STORAGE_RATIO = 2.18, FOOD_DECAY_RATE = 26/month

Brain sees: FOOD_STORAGE_RATIO > 1.0, high decay rate, food being wasted.

Brain outputs: NEED_GRANARY = 0.9 (highest), RESOURCE_BUILDING_URGENCY = 0.5

Orchestrator: argmax = NEED_GRANARY. Scans buildings: granary has foodStorage > 0. Match. Builds granary (+200 food storage).

Example 3: Berry bushes running out, need renewable food

State: Only 2 berry bushes left, food stockpile = 60 falling, FARM_COUNT = 0, GATHER_FOOD = 1.2/s

Brain sees: Low food, low gather speed, no farms yet. LACKING_FOOD is high.

Brain outputs: NEED_FARM = 0.95 (highest), RESOURCE_BUILDING_URGENCY = 0.8

Orchestrator: argmax = NEED_FARM. Scans buildings: farm has toResource >= 0, spawnResourceRange <= 0. Match. Builds farm (converts to 1600HP food node).

Example 4: Trees depleted, need spawner

State: Only 3 trees left, SPAWNER_COUNT = 0, BONUS_COVERAGE_WOOD = 0.8 (existing trees already bonused)

Brain sees: High bonus coverage (sawmill exists) but trees running out. Zero spawners.

Brain outputs: NEED_TREE_NURSERY = 0.9 (highest), NEED_SAWMILL = 0.1 (low, already covered)

Orchestrator: argmax = NEED_TREE_NURSERY. Scans buildings: tree_nursery has toResource >= 0, spawnResourceRange > 0. Match. Builds tree_nursery.

Example 5: Gold refinery needed

State: 6 workers on gold, GATHER_GOLD = 6.0/s, no refinery, GOLD_REFINE_GAP = 6.0 (all gathering wasted)

Brain sees: High gold gather rate but zero refine capacity. GOLD_REFINE_RATIO = 0 (no refineries).

Brain outputs: NEED_GOLD_REFINERY = 0.8 (highest), RESOURCE_BUILDING_URGENCY = 0.6

Orchestrator: argmax = NEED_GOLD_REFINERY. Scans buildings: goldRefinery has goldRefineCapacity > 0. Match. Builds goldRefinery (cap=10, +10% bonus).

Pros and Cons

Unified vs Two-Tier (Previous Option B)

Pros

  • No middleman: EconomyBrain diagnoses the problem AND picks the solution. No lossy signal passing between brains.
  • One brain to train: Single NN, single CSV, single training pipeline. No activation threshold to tune.
  • No duplicated inputs: ResourceBrain needed gather speeds, stockpiles, storage ratios -- all already in EconomyBrain.
  • No ambiguous combinations: Old approach output capability + resource preference independently. Some combos (e.g. REFINE_FOOD, SPAWN_STONE) don't map to any building. Brain could waste effort on impossible outputs.
  • Direct and debuggable: Each output maps to exactly one building type. Easy to inspect what the brain wants.
  • Fallback behavior: If nation doesn't have a building, that need is simply ignored (no candidate matches). No wasted signal.

Cons

  • Larger brain: 34 inputs x 18 outputs vs 24x8 + 20x9. More weights to train, slightly slower inference.
  • Not extensible to new combos: Adding a new building type (e.g. STORAGE_STONE) requires adding a new output to the enum and retraining. Two-tier approach would handle it automatically via capability + resource combination.
  • Building-type coupling: Output names reference specific building types. If game design changes significantly (e.g. removing farms), the enum needs updating.
  • No separation of concerns: Economy macro decisions and building micro decisions are mixed in one brain. May be harder to debug if the brain makes bad economy decisions AND bad building decisions simultaneously.

Direct Building Needs vs Capability + Resource (Alternative)

Pros of Direct Needs (current)

  • Unambiguous: Every output maps to exactly one building type. No invalid combinations.
  • Simpler orchestrator: Just argmax + DB field matching. No capability*resource matrix to resolve.
  • Brain can learn building-specific patterns: E.g. "farm is good when berries run out" is a different pattern from "mill is good when workers lack bonus". These don't share a common "FOOD building needed" signal.

Cons of Direct Needs

  • More outputs: 10 vs 9 (5 capabilities + 4 resource prefs). Grows if new buildings are added.
  • Redundancy: NEED_MILL, NEED_MINE_S, NEED_MINE_G are all "BONUS for resource X". The brain must independently learn the same pattern (low bonus coverage) for each resource.
  • Future nations: If a nation has different resource buildings (e.g. a fishing hut instead of a farm), the output enum stays the same but the name is misleading. Names are types (CONVERT_FOOD), not specific buildings.
