Input Source Flow — Where Inputs Come From
All inputs are normalized floats. Game state objects (Player, Resources, Possession) are sampled each tick and divided by normalization constants. MasterBrain urgencies cascade as inputs to sub-brains.
1. MasterBrain — 32 Inputs (was 29)
Strategic overview of the game state. Removed per-resource lacking (Master only needs total stress). Added game phase, combat efficiency, momentum trends, and tech progress signals.
2. EconomyBrain — 24 Inputs (was 20)
Focused on resource management. Removed military/threat inputs (that's Master's job). Added resource gathering quality signals: storage utilization, refine capacity, and workers on unimproved nodes.

Kept Inputs (17)

  • [0-3] VALUE_FOOD/WOOD/STONE/GOLD — res->getValue(RT) / 1000 — Current stockpile per resource
  • [4-7] GATHER_SPEED_FOOD/WOOD/STONE/GOLD — res->getGatherSpeed(RT) / 10 — Income rate per resource
  • [8] FREE_WORKERS — possession->getFreeWorkersNumber() / 100 — Available labor
  • [9] TOTAL_WORKERS — possession->getWorkersNumber() / 100 — Economic scale
  • [10-13] LACKING_FOOD/WOOD/STONE/GOLD — lackingPerResource[i] / 500 — Closed-loop feedback per resource from WantList
  • [14] ECONOMY_URGENCY — from MasterBrain — Master directive for economic focus
  • [15] WORKER_URGENCY — from MasterBrain — Master directive for worker production
  • [16] EXPAND_URGENCY — from MasterBrain — Master directive for territorial expansion

Removed Inputs (3)

  • ARMY_COUNT — possession->getArmyNumber() / 200 — Not relevant for economy decisions; Master urgencies encode military pressure
  • ENEMY_ARMY_TO_OWN_BUILDING — diffOfCenters(...) — Threat awareness is Master's job
  • ENEMY_SCORE — enemy->getScore() / 1000 — Not directly relevant to economic allocation

New Inputs (8) — Resource Gathering Quality

  • [17] FOOD_STORAGE_RATIO — res->getFoodStorageRatio() — raw (0-2+). If >1, food exceeds storage capacity and decays. Drives "build storage" or "reduce food workers" decisions.
  • [18] GOLD_STORAGE_RATIO — res->getGoldStorageRatio() — raw (0-2+). Gold in storage generates interest. Low ratio = missing free gold income. High ratio = storage is full, no benefit from more.
  • [19] STONE_REFINE_UTIL — res->getStoneRefineUtilization() — raw (0-1). How much of stone refine capacity is used. Low = stone gather speed below refine capacity, wasting potential bonus.
  • [20] GOLD_REFINE_UTIL — res->getGoldRefineUtilization() — raw (0-1). Same for gold refinement. Low utilization = not enough gatherers or refine capacity overbuilt.
  • [21] NO_BONUS_FOOD — possession->getResWithOutBonus(FOOD) / 10 — Workers gathering food without bonus. Less efficient — should be reassigned or bonus buildings built.
  • [22] NO_BONUS_WOOD — possession->getResWithOutBonus(WOOD) / 10 — Same for wood.
  • [23] NO_BONUS_STONE — possession->getResWithOutBonus(STONE) / 10 — Same for stone.
  • [24] NO_BONUS_GOLD — possession->getResWithOutBonus(GOLD) / 10 — Same for gold.
3. BuildingBrain — 14 Inputs (was 14, different composition)
Same total count but completely reworked. Removed granular army/resource inputs that duplicate Master urgency information. Added building counts per parent type so the brain can see what's already built, plus lacking feedback and unit urgency.

Kept Inputs (7)

  • [0] BUILDINGS_COUNT — possession->getBuildingsNumber() / 50 — Total infrastructure
  • [1] WORKERS_COUNT — possession->getWorkersNumber() / 100 — Farm/barracks balance
  • [2] DEFENCE_SUM — possession->getDefenceAttackSum() / 100 — Current defensive structures
  • [3] ENEMY_ARMY_TO_OWN_BUILDING — diffOfCenters(ARMY, enemy, BUILDING, player, 1.f) — Threat proximity drives walls/towers
  • [4] BUILDING_URGENCY — from MasterBrain — General construction priority
  • [5] TECH_URGENCY — from MasterBrain — Technology building priority
  • [6] DEFENCE_BUILDING_URGENCY — from MasterBrain — Defensive structure priority

Removed Inputs (7)

  • FREE_WORKERS — Master urgencies encode worker pressure
  • ARMY_COUNT — Unit urgency from Master captures army needs
  • ATTACK_SUM — Army quality detail not needed for building type selection
  • GATHER_SPEED_FOOD/WOOD/STONE/GOLD — Per-resource income detail belongs in EconomyBrain

New Inputs (7) — Building Type Determination

  • [7] COUNT_OTHER — possession->getBuildingCountByParentType()[OTHER] / 10 — Town centers, houses. Prevents over-building one type.
  • [8] COUNT_DEFENCE — possession->getBuildingCountByParentType()[DEFENCE] / 10 — Walls, towers count.
  • [9] COUNT_RESOURCE — possession->getBuildingCountByParentType()[RESOURCE] / 20 — Farms, mines. High = maybe enough, shift to other types.
  • [10] COUNT_TECH — possession->getBuildingCountByParentType()[TECH] / 5 — Tech buildings. If 0 and tech urgency high, clearly need one.
  • [11] COUNT_UNITS — possession->getBuildingCountByParentType()[UNITS] / 10 — Barracks, stables. Bottleneck for army production.
  • [12] TOTAL_LACKING — totalLacking / 2000 — Resource stress. High = maybe need more resource buildings.
  • [13] UNIT_URGENCY — from MasterBrain — If high, may need more barracks specifically.

New dependency: Possession::getBuildingCountByParentType() returning std::array<unsigned, 5> indexed by ParentBuildingType.

4. UnitBrain — 14 Inputs (unchanged)
No changes proposed. Unit profile matching picks the closest buildable unit; affordability and building availability are handled downstream by WantList and resolveUnit.

All Inputs (14)

  • [0] OWN_INFANTRY_RATIO — possession->getInfantryNumber() / armyCount — Current infantry fraction for rebalancing
  • [1] OWN_CAVALRY_RATIO — possession->getCavalryNumber() / armyCount — Current cavalry fraction
  • [2] OWN_RANGE_RATIO — possession->getRangeNumber() / armyCount — Current ranged fraction
  • [3] ENEMY_INFANTRY_RATIO — enemyPossession->getInfantryNumber() / enemyArmyCount — Counter-picking
  • [4] ENEMY_CAVALRY_RATIO — enemyPossession->getCavalryNumber() / enemyArmyCount — Counter-picking
  • [5] ENEMY_RANGE_RATIO — enemyPossession->getRangeNumber() / enemyArmyCount — Counter-picking
  • [6] ARMY_COUNT — armyCount / 200 — Scale context
  • [7] ENEMY_ARMY_COUNT — enemyArmyCount / 200 — Relative military size
  • [8] ATTACK_SUM — possession->getAttackSum() / 1000 — Army quality
  • [9] DEFENCE_SUM — possession->getDefenceAttackSum() / 100 — Survivability
  • [10] ENEMY_SCORE — enemy->getScore() / 1000 — General threat level
  • [11] WORKERS_COUNT — possession->getWorkersNumber() / 100 — Army-worker balance
  • [12] UNIT_URGENCY — from MasterBrain — Production volume signal
  • [13] ATTACK_URGENCY — from MasterBrain — Offensive vs defensive unit selection
5. MilitaryBrain — 20 Inputs (was 14)
Added the same 6 army composition ratios from UnitBrain. Army composition critically affects whether to attack or defend — cavalry-heavy forces favor raids, range-heavy forces favor positional defense.

New Dependencies Required

  • Possession::getResourcesDestroyed() — float — Exposes private resourcesDestroyed field. Used by MasterBrain for KD_RATIO.
  • Possession::getInCombatNumber() — unsigned (armyNumber - freeArmyNumber) — Used by MasterBrain for IN_COMBAT_RATIO.
  • Possession::getBuildingCountByParentType() — std::array<unsigned, 5> indexed by ParentBuildingType — Computed in ensureReady(). Used by BuildingBrain.
  • MasterBrain internal — prevEnemyScore, prevAttackSum, prevGatherSpeedSum — Tracked in updateHistory() for delta computations.
  • Player::getAvgUnitUpgradeLevel() — float — Average upgrade level across unit types. Used by MasterBrain for TECH_LEVEL.

New Enum File

BrainIndex.h — Replaces magic numbers in nation->actionPrefix[0] etc.

  • enum class ActionBrainIndex : unsigned char { MASTER=0, ECONOMY=1, BUILDING=2, UNIT=3 };
  • enum class OrderBrainIndex : unsigned char { MILITARY=0 };

Normalization Convention

All inputs are rawValue / divisor to keep values roughly in [0, 1] range for the neural network.

  • Counts: /50 (buildings), /100 (workers), /200 (army)
  • Resources: /1000 (values, scores), /10 (gather speeds)
  • Sums: /1000 (attack), /100 (defence)
  • Deltas: same divisor as their base value — keeps scale consistent
  • Ratios: raw (already 0-1), or /3 for ratios that can exceed 1
  • Spatial: raw (diffOfCenters already normalized internally)
