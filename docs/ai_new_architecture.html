New AI Architecture — Complete Plan

Master Brain + Sub-Brains + Want List. Each tick the want list can execute MULTIPLE items (several units, a worker, a building). Unit Brain outputs a 24-float ideal unit profile matched against every unit's metric vector — handles multiple infantry/cavalry/range subtypes naturally. Different brains run at different tick rates (Military=fast, Spatial=on demand, others=1-2/sec). Unit count scaling via military_production output.

System Overview — One Tick
Master Brain evaluates game state → sub-brains generate wants → Want List executes multiple items per tick → lacking feedback. Military Brain is SEPARATE — issues army orders directly (free, no resources).
Tick Rates — Different Brains, Different Speeds
AI compute is expensive — only 1-2 evaluations per second for most brains. Military needs faster response for combat. Spatial brains are triggered on demand (not ticked). This avoids wasting compute on brains that don't need constant updates.

Unit Count Scaling — Multiple Units Per Tick
At 1-2 ticks/second, adding 1 unit per tick takes 10-20 seconds to build an army of 20. That's too slow for an RTS. Solution: Master Brain's military_production output controls how many copies of the matched unit go to the want list.

Key Design Principles

Selective routing — each sub-brain gets only 3-4 relevant master outputs.

Military is separate — army orders are free, go directly to game, NOT through want list. Runs at FAST tick rate for combat responsiveness.

Unit profile matching — brain outputs 24-float ideal vector, matched against ALL units via sqRootSumError. Handles multiple subtypes naturally.

Unit count scaling — count = round(unit_urgency * MAX_UNITS_PER_TICK). Compensates for slow tick rate. One eval can request multiple units.

Multiple items per tick — want list executes everything affordable top-down. Can train 3 units + 1 worker + 1 building in a single tick.

Targeted feedback — only Master (total) and Economy (per-resource) get lacking info. Others stay pure strategy.

Implicit saving — unaffordable high-priority items reserve resources per-tick. No explicit save logic needed.

Multiple spatial brains — 4 specialized brains (Build/Attack/Scout/Expand) with tailored inputs instead of one generic brain. On demand, not ticked.

Tiered tick rates — Military=fast (combat), Strategy brains=slow (1-2/sec), Spatial=on demand. Saves compute where speed doesn't matter.

Want List — Priority Queue Mechanics
Central execution system for resource-spending actions. Persists across ticks with boost/decay. Reservation is per-tick only — no persistent state beyond the list itself.

Single Tick Flow

Step 1: Carry over unexecuted items from last tick (with their current priority).

Step 2: Sub-brains generate new wants with priority scores.

Step 3: Merge — re-requested items get BOOSTED, not re-requested items DECAY, below min_threshold → DROPPED.

Step 4: Sort by priority (highest first).

Step 5: Walk top-down — affordable → EXECUTE, not affordable → RESERVE resources for this tick, continue to next item.

Step 6: Compute lacking resources → feed back to Master (total) and Economy (per-resource) next tick.

Boost / Decay Mechanics

Boost (brain re-requests): new_pri = carry_pri + boost_amount. Item is still wanted → urgency rises. Cap at 1.0.

Decay (brain stops requesting): new_pri = carry_pri - decay_amount. Brain lost interest → urgency falls. Below min_threshold → drop.

Oscillating: Brain requests on some ticks, not others → priority oscillates. Item stays but never dominates. This is fine — genuine uncertainty.

Boost/decay amounts may need to differ per category (buildings decay slower than units — long-term goals vs short-term).

Lacking Resource Feedback

Master Brain: gets total_lacking_sum — single float. Answers "how resource-starved am I overall?" → shift budget toward economy.

Economy Brain: gets lacking_food, lacking_wood, lacking_stone, lacking_gold — 4 floats. Answers "WHICH resource is the bottleneck?" → direct workers to fix it.

Military & Building: get NOTHING. They focus on strategy. The want list resolves resource conflicts for them.

Feedback is 1-tick delayed (computed after execution, fed to brains next tick).

Potential Enhancements

1. Reservation caps — limit reservations to top N items (e.g. 2-3) to prevent cascade lockout where ALL resources get reserved and nothing executes.

2. Reserve shortfall only — reserve (cost - current_stockpile) instead of full cost. Allows partial spending on non-conflicting items.

3. Max queue size per category — cap at e.g. 5 items per type (units/buildings/workers) to prevent unbounded growth from persistent brain requests.

4. Per-category decay rates — buildings might need slower decay (long-term goals) vs units (short-term tactical needs).

5. Context-dependent MAX_UNITS_PER_TICK — instead of magic constant, derive from game state (e.g. number of idle barracks, population headroom). NN output controls fraction of capacity.

6. Income projection — "I earn 30W/tick, need 80W more → ready in ~3 ticks." Could optimize: "safe to spend 20W on a wall and still afford Barracks in 4 ticks."

7. Spatial brain topology sizing — current sizes are estimates. Profile actual inference cost and adjust hidden layer sizes for performance budget.

8. More spatial brains — could add DefendSpatial (where to set up defenses), RallyPointSpatial (where to gather army before attack), etc.

9. Dynamic tick rate — speed up Military Brain during active combat, slow down during peacetime. Save compute when nothing is happening.
