AI Orchestrator — How the AI Works

Each AI player owns one AiOrchestrator that runs 6 neural-network brains every tick. Two phases: action() decides what to build/train, order() decides where to move.

Key Concepts

Neural Network Brains

Each brain wraps a multi-layer neural network loaded from a CSV file. The network is a simple feedforward NN with layers of weights.

  • Brain class holds vector<Layer*>
  • BrainProvider caches and reuses loaded brains
  • CSV files are per-nation (via db_nation::actionPrefix / orderPrefix), enabling different AI personalities per faction
  • Input values are normalized (scores /1000, counts /200, etc.)

Want List

Central priority queue for all resource-spending actions. Brains don't execute directly — they submit wants.

  • Three types: WORKER, UNIT, BUILDING
  • Re-requested items get boosted (x1.15), old ones decay (x0.85), below 0.05 are dropped
  • Sorted by priority, executed top-down while affordable
  • Tracks lacking resources (what couldn't be afforded) — fed back to MasterBrain next tick
  • Can execute multiple items per tick (e.g., 2 units + 1 building if affordable)

Influence Maps & Spatial Brains

The game maintains per-player influence maps at grid resolution. Spatial brains output weights for these maps, not positions directly.

  • 7 maps for general use (AI_MAP_COUNT): buildings, units, attackSpeed, 4x gatherSpeed
  • 3 maps for army use (AI_ARMY_MAP_COUNT): buildings, units, attackSpeed
  • Weights are combined with maps via InfluenceManager::getAreas()
  • Result is filtered by visibility (fog of war) and sorted to find best cells
  • InfluenceMapQuad provides quadtree-based center finding (fallback for army movement)

Feedback Loop

The AI self-corrects via a tick-to-tick feedback loop:

  • Want List computes lacking resources (what it couldn't afford)
  • MasterBrain and EconomyBrain receive this lacking data as input next tick
  • If unit creation fails because no building can deploy it, pendingLackingBuilding is set, boosting building construction urgency

Things To Do

ScoutSpatialBrain

Decide where to send scouts based on fog of war and last-seen timestamps. Inputs: visibility score, game time, urgencies. Outputs: 3 army map weights (inverted by visibility).

ExpandSpatialBrain

Decide where to expand economy. Inputs: resource levels, gather speeds, expand urgency. Outputs: 7 map weights. Uses the currently unused expandPriority from EconomyBrain.

DeploySpatialBrain — Which building to deploy units from

Currently getBuildingToDeploy() returns allPossible.at(0). Should use spatial brain + getBuildingClosestArea() to pick the best-positioned building. Helper already ported.

Tech/Defence building selection

resolveBuilding() for non-RESOURCE types defaults to buildings.at(0). Needs metric matching or a dedicated brain.

Upgrade system

Unit and building level-up logic is fully commented out. Needs: UPGRADE_UNIT / UPGRADE_BUILDING in WantItemType, upgrade decision brain, and re-enabling ActionCenter upgrade commands.

Multi-worker collection

Current system assigns workers individually. Old OrderMaker had a histogram-based group collection system that assigned worker groups to resources simultaneously. Worth porting for better efficiency.

Implement GAME_TIME, KD_RATIO, IN_COMBAT_RATIO inputs

Several MasterBrain inputs are hardcoded to 0.0f with TODO markers. Need actual game state accessors.

Per-sub-army targeting

Currently all sub-armies go to the same target. Each could have different objectives (e.g., one attacks economy, another pins enemy army).

CSV training pipeline

Need to generate/train the neural network CSV files for BuildSpatialBrain (build_spatial.csv, 13→7) and AttackSpatialBrain (attack_spatial.csv, 14→3).
