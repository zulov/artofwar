AI Orchestrator — How the AI Works

Each AI player owns one AiOrchestrator that runs 6 neural-network brains every tick. Two phases: action() decides what to build/train, order() decides where to move.

Key Concepts

Neural Network Brains

Each brain wraps a multi-layer neural network loaded from a CSV file. The network is a simple feedforward NN with layers of weights.

  • Brain class holds vector<Layer*>
  • BrainProvider caches and reuses loaded brains
  • CSV files are per-nation (via db_nation::actionPrefix / orderPrefix), enabling different AI personalities per faction
  • Input values are normalized (scores /1000, counts /200, etc.)

Want List

Central priority queue for all resource-spending actions. Brains don't execute directly — they submit wants, and the WantExecutor turns affordable wants into game commands.

  • Five types: WORKER, UNIT, BUILDING, UNIT_UPGRADE, BUILDING_UPGRADE
  • Re-requested items get boosted with age (boost = 2.0 * age/(age+8), up to +200% with diminishing returns), inactive ones decay (x0.85), below 0.05 are dropped
  • Sorted by priority, capped at 32 items, executed top-down while affordable
  • Tracks lacking resources (what couldn't be afforded) — fed back to MasterBrain (total) and EconomyBrain (per-resource) next tick
  • Can execute multiple items per tick (e.g., 2 units + 1 building if affordable)

Influence Maps & Spatial Brains

The game maintains per-player influence maps at grid resolution. Spatial brains output weights for these maps, not positions directly.

  • 12 maps for general/building use (AI_MAP_COUNT): buildings, units, attackSpeed, 4x gatherSpeed, enemy buildings, enemy units, enemy attackSpeed, res-not-bonus, economy
  • 6 maps for army use (AI_ARMY_MAP_COUNT): buildings, units, attackSpeed, enemy buildings, enemy units, enemy attackSpeed
  • Weights are combined with maps to score map cells
  • Result is filtered by visibility (fog of war) and sorted to find best cells (getBestVisibleAreas)
  • Quadtree-based center finding provides the fallback target for army movement

Feedback Loop

The AI self-corrects via a tick-to-tick feedback loop:

  • Want List computes lacking resources (what it couldn't afford)
  • MasterBrain and EconomyBrain receive this lacking data as input next tick
  • If unit creation fails because no building can deploy it, pendingLackingBuilding is set, boosting building construction urgency

Things To Do

Tech/Defence building selection

resolveBuilding() for non-RESOURCE types defaults to buildings.at(0) (see TODO in code). Needs metric matching or a dedicated brain. Resource buildings already use need-driven selection.

DeploySpatialBrain — which building to deploy units from

WantExecutor::pickDeployBuilding() picks the ready producer closest to a center. A dedicated spatial brain could choose the best-positioned producing building instead.

ScoutSpatialBrain

Decide where to send scouts based on fog of war and last-seen timestamps. No scouting exists yet — the AI plays blind until enemy contact.

ExpandSpatialBrain

Decide where to expand economy using expandPriority from EconomyBrain. Today expansion only influences BuildSpatial weights indirectly.

Retreat & FOW-aware advance

Army has no health-threshold retreat, and the attack-position fallback can send units toward stale/unseen influence centers (TODO in order()). Factor visibility back into target commitment.

Attack outcome tracking

History records "order sent" as SUCCESS, not whether the engagement was won. Tracking real outcomes would let the failure/recency feedback reflect lost battles.

Save/load delta continuity

MasterBrain prev* fields (prevScore, prevEnemyScore, prevUnits, prevResSum, prevGatherSum) are not serialized, so the first tick after load produces false delta spikes.

CSV training pipeline

No GA/training pipeline lives in this repo — brains must be evolved externally. tools/gen_brain_csv.ps1 only emits random untrained weights in the on-disk format.
