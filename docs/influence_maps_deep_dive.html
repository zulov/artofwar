2. Core Mechanics
Every InfluenceMap has the same spatial math even when the game uses it for different meanings.

Grid and Kernel Constants

  • Influence cell size: INF_GRID_FIELD_SIZE = 8 world units.
  • Visibility cell size: VISIBILITY_GRID_FIELD_SIZE = 4 world units.
  • Grid indexing supports the advertised map resolutions 160, 192, 256, 320, 384, 512; it does not require a power-of-two side length.
  • Kernel radius: INF_LEVEL = 4 cells, so the stencil is 9 x 9.
  • Kernel template coefficient: coef = 0.5 for all maps created by InfluenceManager.
  • The 9 x 9 template is allocated once with createTemplateV() and shared by all InfluenceMap instances.

Stored Layers Inside One InfluenceMap

  • rawValues: direct counts or accumulated event strength per influence cell.
  • pendingValues: optional buffer for history maps only.
  • values: kernel-smoothed values used by AI scoring and influence debug view.
  • quadLayers: a 2x2 sum pyramid built from raw values, used only for center finding.
  • center: cached world position of the most representative raw hotspot.

Two Constructor Modes

  • Immediate/snapshot mode: update() adds straight into rawValues.
  • Buffered/history mode: update() adds into pendingValues; reset() applies raw = raw * vanishCoef + pending.
  • History maps in this repo use minimalThreshold = 0.0001 and vanishCoef = 0.5.
  • resetToZero() later drops raw values below the threshold so old noise does not live forever.

Visibility Interaction

  • VisibilityMap runs at double resolution and then downscales by 2x2 blocks.
  • If any cell in a 2x2 visibility block is VISIBLE, the corresponding influence cell is visible.
  • SEEN does not count as visible for influence selection.
  • Unseen influence cells are initialized with FLT_MAX-like error so they sort last automatically.
Kernel template used by every InfluenceMap
template(dx, dy) = 1 / (((dx * dx + dy * dy) * coef) + 1)

kernel[cell] = sum over all raw source cells:
               raw[source] * template(offset from source to cell)

How To Read The Kernel Schema

  1. Start from the two raw cells: one source with value 6, one source with value 4.
  2. The raw-grid colors only distinguish source identity. They do not encode strength ranking.
  3. Each source stamps the same local kernel shape around itself.
  4. For every touched kernel cell, multiply the source value by the template weight for that offset.
  5. If two stamped patches overlap, add both contributions. That is why some cells in the final kernel grid are larger than either source would create alone at that offset.
  6. In the final kernel heatmap, colors are monotonic: blue is lowest, then cyan, green, yellow, orange, and red as values increase.

Example takeaway: the kernel layer is not a separate game event. It is a derived density field built entirely from rawValues. Multiple nearby raw cells create overlapping hills, and the overlap itself can become the most attractive place in the kernel field.

Quad Layer Construction Schema

Before the engine can ask for a center, it first builds the quad pyramid from the raw grid. Each layer is produced by summing 2x2 blocks from the previous layer.

Layer One parent cell represents Example value
raw 8x8exact original cell5 at one hot raw cell
quad 4x4sum of one raw 2x2 block11 = 1 + 2 + 3 + 5
quad 2x2sum of one raw 4x4 region16 for the dominant quadrant

How To Read The Quad Build Schema

  1. Start from the raw grid.
  2. Collapse every raw 2x2 block into one summed parent value.
  3. Repeat the same process on the new layer until the pyramid reaches its coarsest level.
  4. The pyramid stores mass by region, so later center lookup can find the dominant area without scanning every raw cell at every step.

Example takeaway: quad layers do not invent new signal. They are just progressively coarser sums of the raw layer.

Center Computation Schema

Once the quad pyramid already exists, getCenter() starts at the smallest stored quad layer, picks its best cell, then refines that choice through successively higher-resolution layers until it reaches the raw grid. This is a separate step from building the quad layers.

How To Read The Center Schema

  1. Start at the smallest stored quad layer, which is quadLayers[0] in the implementation. In this example that layer has resolution 2x2.
  2. Pick the best cell there, then convert that cell index into the four corresponding children on the next higher-resolution layer, here 4x4.
  3. Repeat that refinement step once more to reach the raw grid, here 8x8.
  4. Once the search reaches the raw grid, compare the final four raw cells directly.
  5. Return the world-space center of the best raw cell, not the average position of the whole winning region.

Example takeaway: quad construction and center selection are separate operations. First the pyramid is built from sums; later the center algorithm starts from the smallest stored quad layer and refines upward in resolution until it reaches a final raw-cell winner.

History Map Schema

History maps add one more layer: pendingValues. Fresh events land there first. Only reset() moves them into rawValues. Until that happens, kernel, quad, and center still read the old raw state.

In this codebase, gatherSpeed[...] and attackSpeed are the two history-map families. Both use vanishCoef = 0.5 and minimalThreshold = 0.0001, so the runtime treats them as fading heatmaps rather than permanent counters.

When And How Often

  • Buffered event writes: irregular. They happen whenever gameplay causes collection or attack history to be recorded.
  • Merge/decay pass: regular. It runs once per second in the default 30-FPS simulation schedule.
  • Kernel rebuild after merge: immediate relative to the merge pass. It happens in the same scheduled history update via finalize().
  • Threshold cleanup: also regular. It is scheduled once per second as a separate history-reset action.
  • Startup forced pass: one-shot. It runs before the first normal game tick so the map system starts from a known state.

How To Read The History Schema

  1. Stage A is the startup case: the map exists, but there is no history yet, so raw, pending, kernel, and center are all empty.
  2. Stage B shows the key timing rule: update() writes into pendingValues, so the map has new information internally, but kernel and quad still cannot see it.
  3. Stage C is the first merge: reset() moves pending into raw using raw = raw * vanishCoef + pending.
  4. Stage D isolates the kernel rebuild after the first merge, because the visible kernel change is a distinct moment in the runtime sequence.
  5. Stage E isolates the pure decay case: no new event arrives, so the existing hotspot simply fades.
  6. Stages F and G then show the same buffering-and-merge sequence again, but now starting from an already decayed raw state.

Detailed Runtime Timeline

  1. A gameplay event happens. For attack history this is typically addAttack(); for economy history it is addCollect(). Those manager functions call InfluenceMap::update() on a history map.
  2. Because the map owns pendingValues, the new value is accumulated into the pending buffer instead of raw.
  3. update() also marks all caches dirty: kernel, quad, center, and min/max. That is important because even though raw did not change yet, the logical state of the map has changed.
  4. Later, on the scheduled history update, InfluenceManager::updateWithHistory() calls reset() on each history map. In the default schedule this happens once per second when the INFLUENCE_OTHER action fires on frame slot 3 of the 30-frame period.
  5. That reset() call is the exact moment when pending is merged into raw and old raw is decayed by 0.5.
  6. At that exact moment raw has changed, pending has been cleared, and all caches are dirty, but kernel is not yet materialized unless someone reads it.
  7. Immediately after that, updateWithHistory() calls finalize(), which triggers updateFromTemp() and therefore forces the kernel cache to rebuild immediately from the new raw state.
  8. Quad and center still rebuild lazily, but they now rebuild from the already-merged raw history, so they can finally see the new hotspot.
  9. If no new event happens before the next scheduled history update one second later, the exact same reset() call still changes the map: pending contributes 0, so raw simply decays by half.
  10. If a later event happens after that decay step, pending fills again, but kernel and center still show the decayed raw state until the following merge.
  11. On a separate scheduled action in the same once-per-second slot, resetHistoryThresholds() calls resetToZero(). Any raw cell below 0.0001 is dropped to zero so stale noise does not linger forever.

Important distinction: finalize() does not merge pending into raw. The merge is done by reset(). finalize() only ensures the kernel cache is rebuilt from whatever raw now contains.

Example takeaway: history maps warm up one merge later, then continue through a repeating cycle: buffer fresh events, merge them into raw, rebuild visible kernel from raw, decay when no events happen, buffer again, merge again. At every point, kernel/quad/center are driven by raw, not by pending.

Why Raw, Kernel, and Quad All Exist

raw keeps the exact counts and exact event locations. This is what special placement APIs such as getAreasResBonus() use, and it is also what the center-finding pyramid is built from.

kernel spreads local signal into neighboring cells, which makes AI placement and target choice less brittle than picking only exact occupied cells.

quad + center builds a coarse-to-fine sum pyramid from the raw layer, starts center search from the smallest stored quad layer, refines that choice through higher-resolution layers, and finally returns the winning raw cell's world-space center. The current implementation does not use the kernel layer for center selection.

pending exists only on history maps. It is a transient write buffer for fresh collect and attack events. Until reset() merges it into raw, the kernel, quad, and center caches still reflect the previous raw state.

That last point matters: a recent cluster of activity can influence placement through the kernel, while getCenter() still points at the strongest raw cell inside the dominant raw cluster.

How a weight turns one map into AI error
if weight > 0:
    error += (((max - value) / (max - min)) * weight) ^ 2

if weight < 0:
    error += (((value - min) / (max - min)) * weight) ^ 2

if weight == 0 or max == min:
    this map contributes nothing

Interpretation of Weight Sign and Magnitude

  • Positive weight means seek high values in that map.
  • Negative weight means seek low values in that map.
  • Larger absolute value means the map matters more because the squared error gets larger.
  • Uniform maps are ignored entirely because max == min and they cannot help discriminate between cells.
  • After visibility masking and error accumulation, the manager keeps only cells whose error is below 10 * numberOfNonUniformMaps, sorts them best-first, and caps the result count at arraySize / 8.
3. Map Inventory
This section describes every concrete map family created by InfluenceManager, not just the AI-facing aliases.

buildingsInfluencePerPlayer

snapshotkernel-consumedBUILDING center

Raw meaning: number of buildings owned by that player in each influence cell. Updated by updateBuildings() after a full reset.

Kernel meaning: blurred building presence. This is what the build and attack spatial brains actually score.

Center meaning: CenterType::BUILDING resolves to this map's raw hotspot, not the kernel hotspot.

AI consumption:

  • Build spatial slot 0 = our buildings, slot 7 = enemy buildings.
  • Attack spatial slot 0 = our buildings, slot 3 = enemy buildings.

Typical use: positive for placing support buildings near an existing base; negative for expanding away from a crowded footprint; positive on the enemy view for aggressive forward placement or target bias.

unitsInfluencePerPlayer

snapshotkernel-consumed

Raw meaning: number of all units in each cell, workers included. Updated by updateUnits() after a full reset.

Kernel meaning: general unit density around each cell.

AI consumption:

  • Build spatial slot 1 = our units, slot 8 = enemy units.
  • Attack spatial slot 1 = our units, slot 4 = enemy units.

Typical use: positive on our map keeps production and defense near population; positive on the enemy map pulls attacks toward likely contact zones; negative can search for open space.

Debug mode UNITS_NUMBER_PER_PLAYER shows the raw map, while UNITS_INFLUENCE_PER_PLAYER shows the kernel-smoothed version.

armyInfluence

snapshotraw-centeredARMY center

Raw meaning: number of non-worker units in each cell. It is updated inside updateUnits() and ignores workers.

This family is not part of the 12-slot build schema or the 6-slot attack schema. Its main job is providing a stable army center for higher-level distance metrics and military planning.

Primary consumers:

  • CenterType::ARMY
  • distance inputs in MasterBrain and MilitaryBrain
  • raw values remain available for center calculation; no dedicated debug mode currently exposes this map

gatherSpeed[FOOD, WOOD, STONE, GOLD]

buffered historykernel-consumed

Raw meaning: decayed recent collection intensity for one resource type. Updates arrive through addCollect(), which writes to pendingValues at the worker's influence cell.

On updateWithHistory(), each cell performs raw = raw * 0.5 + pending; then pending is cleared and the kernel cache is rebuilt lazily.

Threshold cleanup later removes raw values below 0.0001 so old collection trails disappear completely.

AI consumption:

  • Build spatial slots 3, 4, 5, 6 map to food, wood, stone, gold respectively.

Typical use: positive means "place near active gathering for that resource"; negative means "avoid the currently busy gather lanes".

attackSpeed

buffered historykernel-consumedBATTLE center

Raw meaning: decayed recent combat activity for a player. Updates arrive through addAttack(), which is called by attack and projectile code paths.

Like gather maps, it uses pendingValues, vanishCoef = 0.5, and threshold pruning. This makes it a battle heatmap, not a direct unit count.

AI consumption:

  • Build spatial slot 2 = our attack history, slot 9 = enemy attack history.
  • Attack spatial slot 2 = our attack history, slot 5 = enemy attack history.
  • CenterType::BATTLE resolves to this map for the current player.

The code comment is accurate: BATTLE is only an alias to the player's attackSpeed map. It is not a shared cross-player battle center.

economyInfluence

event accumulatorECON center

Raw meaning: recent collection activity written directly by addCollect() at the worker's cell. Unlike gather maps, this family has no pending buffer and no decay coefficient.

Instead, it accumulates until updateQuadOther() resets it. In practice this makes it a short-window "where collection recently happened" map.

AI consumption:

  • Build spatial slot 11.
  • CenterType::ECON.
  • debug mode ECONOMY (raw).

Typical use: positive weight attracts storage, refine, or defense toward the current economy core; negative weight pushes buildings away from the economy if the placement class wants empty space.

resNotInBonus[RESOURCE]

snapshotraw maxima APIspecial placement

Raw meaning in the current implementation: resource nodes currently being worked by this player's workers, for one specific resource type.

How it is built: during updateUnits(), if a worker is in COLLECT state and still has a live target, the code marks the resource entity's own influence index. So the peaks are on the resource cells, not on the worker cells.

Important naming note: despite the member name, this map does not currently check whether the resource has a bonus. It records active collection at resource nodes. The "without bonus" concept does exist elsewhere in the codebase, for example in Possession::ensureResWithoutBonus(), but that check is not used when filling this map family.

Primary consumer: getAreasResBonus(resourceId, player), which returns getRawMaxIdxs() from this map. That means the special path deliberately uses the exact raw peaks rather than the kernel-smoothed field.

There is an Environment::getPosToCreateResBonus() helper for this path, but the current generic build flow uses the 12-weight build-spatial output instead of calling that helper directly.

resNotInBonusAny

snapshotbuild-spatial only

Raw meaning: the union of all four resNotInBonus[RESOURCE] families. If any worker is collecting from any resource node, this map gets a mark at that resource's influence cell.

This family exists because the build-spatial brain only has one general slot for this resource-activity signal rather than four separate slots.

AI consumption:

  • Build spatial slot 10 only.

Typical use: positive weight is a broad "place my resource-support building near current resource activity" signal, independent of resource type.

VisibilityMap

visibilityselection mask

Resolution is double the influence-grid resolution. Every unit and building contributes its sight radius through update(position, sightRadius). After all updates, finish() expands those radii into per-cell visibility states.

Cell states are:

  • NONE = 0 - never seen / currently hidden
  • SEEN = 1 - explored before, not currently visible
  • VISIBLE = 3 - currently visible

reset() demotes VISIBLE to SEEN by applying a bit-mask; it does not wipe exploration history. For influence selection, the map downsamples 2x2 visibility cells into one influence cell, and only VISIBLE counts.

Main consumers:

  • removeUnseen() seeds the error array for getAreas()
  • getVisibilityScore() for AI metrics
  • runtime hide/show of units, buildings, and resources

How Build Placement Uses the Ordered Candidate List

WantExecutor::findPosToBuild() calls the build-spatial brain, passes its 11 weights to Environment::getPosToCreate(), and receives an ordered list of influence cell indexes from InfluenceManager::getAreas().

The environment does not place the building directly on the influence cell center. Instead it:

  • converts each influence index to a world-space center,
  • searches nearby main-grid centers around that influence area,
  • checks validateStatic() for footprint fit,
  • checks current visibility,
  • returns the first valid build position.

This is why influence maps choose promising areas, not exact tile placements.

How Army Targeting Uses Centers

MilitaryBrain produces directional pressure between ARMY, ECON, BUILDING, and BATTLE centers. MilitaryCommandCalculator combines those pressures with each unit's proximity to select a target center.

Each selected bucket moves directly to that center. Fog of war is intentionally ignored for army center targets, so an enemy center may be stale or hidden.

These examples are aligned with the current implementation and test cases. The three schema cards above cover the low-level data flow; the examples below show how those mechanics change ranking and AI behavior.

Example 1 - Positive and Negative Weights Combined

This mirrors the regression test that combines "seek high buildings" with "seek low gathering". The real test map is larger than the four cells shown here, so the building map's minimum is 0 because the untouched cells are empty.

Step by step:

  1. The buildings map is scored with a positive weight, so high building values get low error. Because min = 0, cells with building value 0.1 are penalized more than they would be in a four-cell-only toy example.
  2. The gather map is scored with a negative weight, so high gather values get high error. Cell 1 is hurt by gather congestion even though its building support is perfect.
  3. The two errors are added cell by cell. No map "wins" on its own; the final ranking comes from the sum.
  4. Cell 0 wins because it has the best building value and zero gather congestion. Cell 3 loses because it is weak on the building map and crowded on the gather map at the same time.

Other untouched cells in the real test map sit at building value 0, so their building error is 0.81. That keeps the intentionally bad empty cells behind the shown candidates.

Example 2 - History Decay in attackSpeed or gatherSpeed

Suppose one cell currently has raw value 6 from previous fights, and this tick it receives 4 more attack value in the pending buffer.

Before reset() Formula After reset()
raw = 6, pending = 4 raw = raw * 0.5 + pending raw = 7

Step by step:

  1. Before the merge, the map's visible state is still based on raw value 6. The fresh 4 lives only in pending.
  2. When reset() runs, half of the old history survives and the fresh event is added on top: 6 * 0.5 + 4 = 7.
  3. Kernel rebuild now uses raw value 7, so the local history hill becomes stronger than it was one tick earlier.
  4. If the following tick has no fresh event, the same cell decays to 7 * 0.5 = 3.5. The hotspot fades, but it does not disappear instantly.

This is exactly why attack and gather maps behave like fading heatmaps instead of permanent counters.

Example 3 - Visibility Downsample Rules

The visibility grid is twice as fine as the influence grid. One influence cell corresponds to one 2x2 visibility block.

2x2 visibility block Influence cell visible? Reason
VISIBLE, NONE, NONE, NONEyesany visible child makes the parent visible
SEEN, SEEN, SEEN, SEENnoseen-only does not count as currently visible
NONE, NONE, NONE, NONEnothe parent cell is unseen and gets max-float error

Step by step:

  1. The visibility system first decides visibility at the finer fog-of-war resolution.
  2. Then every 2x2 visibility block is collapsed into one influence-cell visibility flag.
  3. If any child is VISIBLE, the influence cell is eligible for map scoring.
  4. If all children are only SEEN or NONE, the influence cell is treated as unavailable for current area selection.

This behavior is locked down by VisibilityMapTest.cpp. For area selection, only current visibility matters.

Example 4 - Why getCenter() Can Differ from Kernel Intuition

A test case sets raw cell 4 to 1 and raw cell 5 to 2. getCenter() returns the world center of cell 5.

Step by step:

  1. The quad pyramid sums raw mass into larger and larger blocks, so the search begins with regional dominance rather than exact-cell dominance.
  2. Once the best cell on the smallest stored quad layer is known, the search refines that choice into the corresponding child block on the next higher-resolution layer.
  3. At the last step, only the final four raw candidates are compared directly, and raw cell 5 wins because its raw value is larger than raw cell 4.
  4. Even if kernel smoothing would make neighboring cells look equally attractive, the returned center is still anchored to the strongest raw winner.

Example 5 - Specialized Bonus Placement vs Generic Build Spatial

If three workers collect wood from three separate trees, resNotInBonus[WOOD] gets three raw peaks on those resource cells.

Step by step:

  1. During updateUnits(), each collecting worker marks the resource entity's influence index, not the worker's own current position.
  2. The specialized helper getAreasResBonus(WOOD) reads the raw peaks directly, so it answers a very narrow question: "where are active wood collection nodes right now?"
  3. The generic build-spatial path uses the broader resNotInBonusAny map, but combines it with 11 other weighted maps.
  4. That broader path can choose to sacrifice the strongest resource-activity hotspot if economy, danger, congestion, or enemy pressure make another visible area a better overall build location.

Example 6 - Startup History Map With Fresh Pending Data

This is the exact case that often feels surprising when reading the code for the first time.

Moment raw pending What getKernel() sees What getCenter() sees
after construction / forced startup update, no events yet00all zerosno center
after first addAttack() or addCollect()0non-zerostill all zerosstill no center
after the next history reset()non-zero0non-zero kernel hillvalid center

Step by step:

  1. The forced startup influence pass does run, but it only merges whatever is already in pending. At true startup that is nothing, so raw stays zero.
  2. The first real collect or attack event writes into pending only.
  3. If some AI query asks for kernel or center before the next history merge, it still reads the old raw state, which is empty.
  4. After the next scheduled merge, the map becomes "live" for kernel scoring, quad search, and center lookup.

Common Gotchas

  • valueThresholdDebug is only for rendering. The thresholds like 2, 5, and 40 do not change simulation logic.
  • BATTLE is a per-player battle-history alias. It is not a shared global battlefield center.
  • Uniform maps do not help placement. If every cell in a map has the same kernel value, cumulateErrors() returns false and that map contributes nothing.
  • Visibility filtering happens before map scoring. Unseen cells start with huge error, so even a perfect influence score elsewhere cannot normally beat them.
  • getRawMaxIdxs() and getKernelMaxIdxs() are top-10 helpers. They sort descending and return nothing if the maximum value is not above 0.5.
  • Center queries use raw data. If you expect the center of a blurred region, that is not what getCenter() currently computes.
  • Generic building placement and specialized bonus placement are separate paths. The generic path uses the 12-weight view; the special path uses raw maxima from resNotInBonus[RESOURCE].
